# Agent-Based Assessment: Issues, Fixes, and Comparison with Published Systems
## Meeting March 23, 2026

---

## 1. What We Tried

Ran Claude Sonnet 4.6 (Computer Use API) against 4diac IDE 3.0.2 in Docker on **Windows x86_64** (native, no QEMU). 4 tasks, 30 steps each, 120 screenshots total.

**Result:** 0/4 tasks completed. 12 usability issues found, but all from surface-level interaction (Welcome screen, menus). Agent never reached the actual IDE workspace.

---

## 2. Three Blockers — Why the Agent Failed

### Blocker 1: Welcome Screen Trap (all 4 tasks)

The agent spent 15-20 of its 30 steps trying to close the Welcome/Tutorials tab. Clicking at `(101, 100)` hit the zoom toolbar instead of the tab X button.

From Task 1 report:
> "I see the Welcome tab seems to have disappeared from the tab bar but the content is still showing. I also see a tooltip 'Set zoom level to fit the page'. The tab appears to have been closed but the Tutorials content is still visible."

> **USABILITY ISSUE #2**: "The Welcome tab appeared to close but the content panel is still showing — this creates confusion about whether the tab was actually closed."

**Impact:** Every task burned 50-66% of its step budget before reaching the IDE workspace.

### Blocker 2: Screenshot-to-Coordinate Mismatch (Tasks 2, 3)

The container display runs at **1920x1080**, but the Claude API receives a downscaled screenshot (~770x580). The agent reasons about pixel positions in the small image, but xdotool executes at full resolution. The agent clicked `x=364` twelve times trying to hit "Window" menu — and got "Project" every time.

From Task 2 report:
> "The cursor is at x:366, y:26. Looking at the screenshot image — the screenshots are 770x580 pixels of content but the actual display coordinates might be different. If the screen is 1920px wide, then the coordinate scale factor is 1920/770 ≈ 2.49."

> "I keep getting the Project menu when clicking at x=364. Looking at the screenshot, the 'Project' menu item in the menu bar appears highlighted/selected."

**Impact:** Tasks 2 and 3 spent 24+ steps clicking the wrong menu item. The agent correctly identified "Window" visually but couldn't translate that to the right coordinate.

### Blocker 3: Small Click Targets in Dialogs (Task 4)

Task 4 was the most successful — Ctrl+N actually opened the New Wizard dialog (proving keyboard shortcuts work on native x86_64). But the agent couldn't expand the "4diac IDE" tree node because the expand triangle is ~8 pixels.

From Task 4 report:
> "I can see the '4diac IDE' option. Let me click on the triangle/arrow next to it to expand it."
> "The tree is not expanding. Let me try double-clicking on '4diac IDE'."
> "The tree is still not expanding."

**Impact:** Agent reached the dialog (progress!) but couldn't navigate it.

---

## 3. Proposed Fixes

| # | Fix | Effort | Impact |
|---|-----|--------|--------|
| 1 | **Pre-dismiss Welcome screen** via xdotool before agent starts | Low | Eliminates Blocker 1 — saves 15+ steps |
| 2 | **Lower resolution** to 1024x768 (match what Claude sees) | Low | Eliminates Blocker 2 — coordinates align |
| 3 | **Pre-load example project** in workspace | Low | Agent starts with content to explore |
| 4 | **Increase step budget** to 50+ | Trivial | More room for exploration |
| 5 | **Add keyboard fallback instructions** to system prompt | Low | Agent uses Alt+W, arrow keys when clicks fail |
| 6 | **Add accessibility tree extraction** (AT-SPI) | High | Gives agent structured element map like web agents have |

Fixes 1-5 are engineering changes. Fix 6 is a research contribution suitable for the bachelor thesis.

## 4. How Other Systems Solve These Problems

### 4.1 UXAgent (CHI EA '25) — Web-Based, DOM-Aware

**Their approach:** Agent reads the HTML DOM, not screenshots.

> **p.3, Section 3.3:** "The Universal Browser Connector uses **manually crafted policies to parse the raw HTML** into a simplified version of HTML as the model's observation space and use auto-generated action space that can be easily generalized to different websites."

**Why it works:** The agent knows every button's name, type, and position from the DOM tree. It clicks `search_button`, not `(364, 26)`. No coordinate guessing.

> **p.3, Section 3.3:** "The simplified HTML is a tree structure that contains the visible information of the web page, such as the title, the product list, the product detail, and the shopping cart, **while trimming all unnecessary information**, such as the CSS and JavaScript code."

**Why we can't do this:** 4diac IDE is a Java/SWT desktop application — there's no HTML DOM. The equivalent would be extracting the accessibility tree (AT-SPI on Linux), which is Fix 6.

**Their action space:**
> **p.3, Section 3.3:** "Action space is a set of task-agnostic actions generated by the parser, such as 'click', 'type', and 'back', that can be easily generalized to different websites."

Our agent uses the same actions (click, type, key) but targets pixel coordinates instead of named elements.

### 4.2 UXCascade (2026, ETH Zurich + Adobe) — Structured State Summaries

**Their approach:** Agent receives a structured state summary at each step.

> **p.8, Section 5.1:** "At each step, the agent receives a **structured state summary derived from the current DOM and page visuals**. This summary includes an indexed map of interactive elements, tab-level metadata, and an annotated screenshot."

**Why it works:** The agent gets both a screenshot AND a labeled map of every interactive element. It's like having a GPS overlay on a satellite photo.

> **p.8, Section 5.1:** "The agent then selects the next action from a predefined task-agnostic set (e.g., click, scroll, type), **based on its internal reasoning**. After executing the action, the agent logs the outcome, updates the interaction history, and records any errors."

**Their step budget is similar to ours (25 steps)** — but they don't waste steps on Welcome screens because the website is already loaded and ready.

> **p.8, Section 5.1:** "Simulation Agents operate with **temperature=1.0** to encourage behavioral diversity across runs, capped at **25 steps per agent** to balance realism and cost."

**Key difference from our setup:**

| Aspect | UXCascade | Our Setup |
|--------|-----------|-----------|
| Perception | DOM + annotated screenshot | Raw screenshot only |
| Element identification | Indexed map with labels | Visual recognition from pixels |
| Starting state | Website loaded, ready to test | Welcome screen + Updates popup |
| Coordinate system | Element IDs (no coordinates) | Pixel coordinates (error-prone) |

### 4.3 Multimodal Insight Summarization (ICMI '24) — No Agent, Real Users

**Their approach:** Don't use agents at all. Record real humans, then use LLMs to summarize.

> **p.7, Section 4.1:** "During each UX test session, the UserTesting platform concurrently records **four different data streams**: events/interactions, UI design, screen capture, and think-aloud audio."

> **p.8, Section 4.2:** "The platform combines the event/interaction and DOM data streams into a **'behavioral transcript'** that describes the participant's actions throughout the session in natural language."

**Why it works perfectly:** No coordinate problems, no scaling issues, no Welcome screen. Real humans operate the software; LLMs just analyze the recording.

**Their scale proves the approach:**
> **p.9, Section 5:** "Between August 30, 2023 and April 30, 2024, UX researchers created **75,464 UX studies** eligible for insights summarization. [...] **3,821 of them (34.6%) created 56,830 insights summaries.**"

**Their future work section explicitly calls for what we're doing:**
> **p.10, Section 6:** "Future work should examine whether leveraging **inherently multimodal LLMs such as Gemini** could improve performance. Can multimodal LLMs automatically generate a multimodal transcript from **screen capture video and think-aloud audio** without having to construct an intermediate verbal-behavioral transcript?"

**We are already doing this.** Our Gemini video assessment is exactly their proposed future work — and it achieves ~80% hit rate on the ground truth issues.

### 4.4 VideoWebArena (ICLR 2025) — Video Helps Observation, Not Action

**Their key finding that validates our approach:**

> "Long-context models perform **worse** with video tutorials than without them, showing a **5% performance decrease** on WebArena tasks and **10.3% decrease** on VisualWebArena tasks."

**What this means for us:**
- **Watching video to identify issues = works** (our Phase 1, Gemini ~80% hit rate)
- **Watching video to learn how to use software = doesn't work** (their skill retention finding)
- This maps exactly to our results: **Phase 1 (video observation) succeeds, Phase 2 (agent interaction) struggles**

The performance gap is massive:
> Best model factual retention QA: **45.8%** vs human **79.3%**
> Best model task success: **13.3%** vs human **73.9%**

Our agent's ~0% task completion on 4diac IDE is consistent with this — desktop environments are even harder than web.

### 4.5 CHI 2024 — UI Mockup Feedback — LLM Critique Has Limits

**Their precision/recall numbers set expectations:**

> GPT-4 Precision: **0.603** | Human Expert Precision: **0.829**
> GPT-4 Recall: **0.380** | Human Expert Recall: **0.336**
> GPT-4 F1: **0.466** | Human Expert F1: **0.478**

> "52% of GPT-4 suggestions rated as 'Accurate', 19% 'Partially Accurate', **29% 'Not Accurate'**"

**Critical finding about iteration:**
> "Feedback quality **decreased** in utility over design iterations" — first critique is useful, subsequent ones less so.

**Relevance:** Our Gemini video assessment likely has HIGHER precision than their static mockup approach because video provides richer temporal evidence. Their 52% accuracy on static images vs our ~80% hit rate on video supports the claim that **video > static images for usability assessment**.

## 5. The Fundamental Difference: Structured vs Pixel-Based

| Capability | Web Agents (UXAgent, UXCascade) | Our Desktop Agent |
|-----------|--------------------------------|-------------------|
| **Sees** | HTML DOM tree + screenshot | Screenshot only |
| **Knows elements** | Yes — every button has a name and ID | No — must visually OCR text from pixels |
| **Click target** | Element ID (`click #search_button`) | Pixel coordinate (`click 364, 26`) |
| **Coordinate errors** | Impossible — targets are semantic | Common — scaling, offset, small targets |
| **Starting state** | Controlled (page loaded) | Uncontrolled (Welcome screen, popups) |

From OSWorld (NeurIPS 2024): best desktop agent achieves **22%** task success vs web agent benchmarks of **40-70%**. Desktop is fundamentally harder because there's no DOM.

## 6. Positioning for Report

### What to claim:

1. **Phase 1 (video assessment) is the primary contribution** — Gemini + video achieves ~80% hit rate against published ground truth. This is consistent with Multimodal Insight Summarization's real-world deployment and implements their explicitly stated future work direction.

2. **Phase 2 (agent interaction) is a proof-of-concept** that reveals the gap between web and desktop agent capabilities. The 3 blockers we documented (Welcome trap, coordinate mismatch, small targets) are instances of known research challenges documented in OSWorld, WorldGUI, and SeeAct.

3. **The ideal methodology combines both** — video assessment for comprehensive issue detection, agent interaction for discovering interactive/onboarding issues that video can't reveal (like our finding that the Welcome screen is a trap).

### Quote to use in report conclusion:

From UXCascade **p.4, Section 3.3:**
> E4: **"I don't think the AI could ever replace a human study. But it could be one more data point, a proxy that we run through to see what it says."**

This perfectly frames our work: AI usability assessment is not a replacement for human studies — it's a complementary, scalable first pass.
